# PDFS 클라우드 전환 설계

> v1 로컬 실행형 → v2 서버형 모니터링 구조  
> 최종 수정: 2026-06-14 · 상태: v2 구현 완료

---

## 1. v1과 v2의 차이

### v1 로컬 실행형

```text
사용자 PC에서 HPDFS.exe 실행
  → smartctl로 내 PC 디스크 SMART 수집
  → 로컬 모델로 고장 위험도 계산
  → 로컬 화면에서 결과 확인
```

v1은 배포와 사용이 쉽지만, 여러 PC를 중앙에서 관리하기 어렵다.

### v2 서버형 구조

```text
[각 PC]
Agent 실행
  → smartctl로 SMART 수집
  → Backend API로 전송

[서버]
FastAPI
  → RandomForest 모델 예측
  → SMART 규칙 기반 보정
  → PostgreSQL 저장

[브라우저]
Streamlit Dashboard
  → 전체 디스크 상태 확인
```

v2는 여러 PC의 저장장치 상태를 중앙에서 관리할 수 있는 구조이다.

---

## 2. 핵심 제약

브라우저는 보안상 사용자의 로컬 저장장치 SMART 정보를 직접 읽을 수 없다. 따라서 로컬 PC에서 실행되는 Agent가 필요하다.

```text
브라우저 직접 수집: 불가능
Agent 기반 수집: 가능
```

## 3. v2 아키텍처

```text
[Local PC]
  agent.py
  smartctl
      ↓ HTTP POST

[Nginx]
  /api/* → FastAPI Backend
  /     → Streamlit Frontend
      ↓

[FastAPI]
  storage_best_model.pkl 로드
  storage_threshold.json 로드
  RandomForest 예측
  SMART 규칙 보정
      ↓

[PostgreSQL]
  disks
  diagnosis_logs
      ↓

[Streamlit]
  대시보드
  디스크 목록
  디스크 상세
  진단 이력
```

---

## 4. Docker Compose 구조

```text
docker-compose.yml
  services:
    postgresql
    backend
    frontend
    nginx
```

| 서비스 | 역할 |
|--------|------|
| postgresql | 진단 결과 및 디스크 상태 저장 |
| backend | FastAPI 진단 API 및 모델 예측 |
| frontend | Streamlit 대시보드 |
| nginx | API와 Frontend 요청 라우팅 |

실행 주소:

```text
http://localhost
```

---

## 5. Nginx 라우팅

```nginx
location /api/ {
    proxy_pass http://backend:8010;
}

location / {
    proxy_pass http://frontend:8501;
}
```

## 7. CI/CD 구조

GitHub Actions를 이용해 main 브랜치 push 시 Docker 이미지를 빌드하고 배포할 수 있는 구조를 설계하였다.

```text
git push
  → GitHub Actions
  → Docker image build
  → Docker Hub push
  → 서버 배포
```

## 8. v2 장점

| 항목 | v1 | v2 |
|------|----|----|
| 실행 방식 | EXE 로컬 실행 | Docker 기반 서버 실행 |
| 데이터 저장 | 로컬 CSV | PostgreSQL |
| 여러 PC 모니터링 | 어려움 | 가능 |
| 모델 교체 | EXE 재배포 필요 | 서버 모델 파일 교체 |
| 대시보드 접근 | 실행 PC에서만 | 브라우저에서 접근 |
| 확장성 | 낮음 | 높음 |
